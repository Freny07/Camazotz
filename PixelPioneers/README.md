## Intentional UX Issues Introduced

Issue 1
Type: User Interface (UI) / Design Consistency 
What was done: Applied non-uniform layouts across the application with inconsistent alignments, varying padding values, and a lack of cohesive design principles.
 Why this was done: To test the developer's ability to identify "Visual Debt" and implement a unified CSS or XML layout system.
 Expected User Difficulty: High. Users will find the app visually jarring and unprofessional, making it harder to focus on the educational content.



Issue 2
Type: User Experience (UX) / Accessibility 
What was done: Created structural irregularities by using ambiguous labels for buttons, leading to vague and confusing screen transitions. 
Why this was done: To demonstrate the importance of intuitive labeling and "Information Architecture" in software development. 
Expected User Difficulty: High. Users will struggle to navigate the app because they cannot predict what a button will do before clicking it.

Issue 3
Type: Documentation / UI Semantic Error 
What was done: Named UI components and buttons with internal or technical jargon that does not explain their actual functionality. 
Why this was done: To highlight the need for user-centric design and the importance of clear, functional naming conventions in the UI. 
Expected User Difficulty: Moderate. While the app might technically work, the learning curve is unnecessarily steep because the interface is not self-explanatory.
Issue 4
Type: Layout / Visual Alignment 
What was done: Items within the Calculator.xml and other files were intentionally misaligned, causing elements to overlap or appear off-center. 
Why this was done: To practice fixing constraint-based layouts and ensuring the UI remains stable across different screen sizes. 
Expected User Difficulty: Low to Moderate. It primarily affects the aesthetic and perceived quality of the app, though overlapping buttons may prevent clicks.



# Functional Bugs / Issues (If Any)

Issue 1
Type: Functional / Logical Error 
What was done: In Calculator.xml, the mathematical operators were programmed to fail or return incorrect values upon execution. 
Why this was done: To challenge the student to debug core business logic and verify that backend arithmetic functions map correctly to frontend button triggers. 
Expected User Difficulty: Critical. The primary tool of the app (the calculator) is rendered useless for actual calculation.

Issue 2
Type: Navigation / Routing Error 
What was done: In the Splashscreen file, the "Back" logic was removed or broken, preventing the user from reverting to the calculator. 
Why this was done: To simulate a "Dead End" navigation bug and teach the importance of stack management in mobile/web routing. 
Expected User Difficulty: Moderate. Users will feel "trapped" on a specific screen, forcing them to restart the application to continue.
Issue 3
Type: Functional / Content Mapping Error 
What was done: The buttons on the Learning Page were programmed to trigger vague or entirely unsuitable results that do not correlate with the arithmetic topics being discussed. 
Why this was done: To train the developer in verifying "Content-Function Integrity"—ensuring that the information displayed to a learner actually matches the interactive elements they engage with. 
Expected User Difficulty: High. Students attempting to learn specific arithmetic operators will become confused or misinformed when the app provides irrelevant feedback or incorrect data.
